# FIREWOLF Nuxt4移行 技術選定根拠書

## 1. 概要

本資料は、FIREWOLFをNuxt2からNuxt4に移行するにあたっての技術選定の根拠を詳しく説明したものです。

## 2. UIフレームワーク選定

### 最終決定: @nuxt/ui (Tailwind CSS + Headless UI)

#### 選定プロセス
1. **現行技術の分析**
   - Buefy (Vue 2用、Vue 3非対応)
   - Bulma CSSベース
   - 独自カスタマイズが豊富

2. **候補技術の評価**
   - @nuxt/ui (Tailwind CSS + Headless UI)
   - Tailwind CSS + 独自コンポーネント
   - @oruga-ui (Buefyの後継)
   - Bulma + カスタムVue 3コンポーネント
   - PrimeVue 4

#### 評価基準と結果

| 評価項目 | 重要度 | @nuxt/ui | Tailwind+独自 | Oruga | Bulma+カスタム | PrimeVue |
|----------|--------|----------|---------------|--------|---------------|----------|
| 開発効率 | ★★★ | A | B | C | D | B |
| 現行デザイン維持 | ★★★ | A | A | A | S | C |
| メンテナンス性 | ★★★ | S | A | B | C | S |
| Nuxt4対応 | ★★★ | A* | S | S | S | S |
| 学習コスト | ★★ | A | B | C | D | B |
| バンドルサイズ | ★★ | A | S | S | A | C |
| コミュニティ | ★ | S | S | C | - | S |

*Nuxt4対応は要確認

#### 決定理由

1. **開発効率の最大化**
   - Tailwind CSSに習熟済みのため、開発スピードが向上
   - @nuxt/uiの豊富なコンポーネントを活用可能

2. **長期的な安定性**
   - Nuxt公式サポートによる継続的なメンテナンス
   - TypeScript完全対応
   - モダンな技術スタック

3. **現行デザインの完全再現**
   - 現行のカスタム色体系を完全移植可能
   - Tailwind CSSの柔軟性により細かなカスタマイズに対応
   - 人狼ゲーム固有のUI要件を満たせる

4. **技術負債の解消**
   - Vue 2 → Vue 3への移行
   - レガシーなビルドツールからViteへの移行
   - モダンな開発体験の実現

## 3. 状態管理選定

### 決定: Pinia

#### 選定理由
- Vuexの公式後継
- Vue 3 Composition APIとの親和性
- TypeScript完全対応
- 既存のVuexコードからの移行が比較的容易

## 4. バリデーション選定

### 決定: vee-validate v4

#### 選定理由
- 現行のvee-validate v3からの移行
- Vue 3完全対応
- Composition API対応
- TypeScript対応

## 5. その他の技術選定

### CSS処理
- **Tailwind CSS**: UIフレームワークとして採用
- **PostCSS**: Tailwindの処理に使用

### 開発ツール
- **Vite**: Nuxt 4標準のビルドツール
- **ESLint 9+**: 最新のリンティング
- **Prettier 3+**: コードフォーマッティング
- **TypeScript 5.7+**: 型安全性の確保

### テスト
- **Vitest**: 単体テスト（Viteネイティブ）
- **Playwright**: E2Eテスト

### PWA
- **@vite-pwa/nuxt**: Vite対応のPWAモジュール

## 6. 技術選定のリスク評価

### 高リスク要素

#### @nuxt/uiのNuxt 4対応状況
- **リスク**: Nuxt 4での動作保証が未確認
- **対策**: 
  - Nuxt 4 RC版での動作検証を最優先で実施
  - 万が一非対応の場合は「Tailwind CSS + 独自コンポーネント」に切り替え

#### デザイン再現の完全性
- **リスク**: 現行デザインを100%再現できない可能性
- **対策**:
  - プロトタイプ段階で主要コンポーネントの再現度を検証
  - カスタムCSS/コンポーネントでの補完を準備

### 中リスク要素

#### 開発者の学習コスト
- **リスク**: @nuxt/uiの学習に時間を要する可能性
- **対策**: Tailwind CSS習熟済みのため最小限に抑制可能

#### パフォーマンスへの影響
- **リスク**: 新しい技術スタックでのパフォーマンス劣化
- **対策**: 段階的な移行とベンチマーク測定

## 7. 代替計画

### Plan B: Tailwind CSS + 独自コンポーネント
@nuxt/uiが使用不可の場合の代替案
- **メリット**: 完全な制御、最小バンドル
- **デメリット**: 開発工数増加、メンテナンス負荷

### Plan C: @oruga-ui
Buefyに近い操作性が必要な場合
- **メリット**: Buefyからの移行が容易
- **デメリット**: Tailwind CSSの利点を活かせない

## 8. 結論

**@nuxt/ui (Tailwind CSS + Headless UI)** を採用することで、以下を実現：

1. **開発効率の向上**: Tailwind CSS習熟度を活用
2. **長期安定性**: Nuxt公式サポート
3. **デザイン完全再現**: カスタム色の完全移植
4. **モダンな開発体験**: TypeScript + Composition API + Vite

リスク要素も特定済みで、適切な対策により成功確率を最大化できる技術選定となっています。
